# MindOutline

## 1. サービス概要

### 1-1. このサービスはどんなサービスか

**MindOutline** は、断片的なメモをツリー構造で整理し、日報や相談文などの文章化まで支援する思考整理アプリです。  
言語化に悩んだ実体験から生まれ、毎日無理なく使い続けられることを重視しています。

### 1-2. 誰の、どんな課題を、どう解決するか

日報や報告を書くときに「何を書けばいいか分からない」と止まってしまう社会人（特に事務職・IT初級者、発達特性により思考整理に負担を感じやすい方）に向けて、

- キーワードメモの入力
- ツリー構造による思考整理
- AIによる文章生成

という流れを提供し、思考整理と言語化の負担を軽減します。

## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情

私は前職で事務補助として働いていた頃、  
日報を書くのに極端に時間がかかるという悩みを抱えていました。

業務内容は頭の中にあるのに、  
文章にまとめることができず、気づけば周囲より大幅に遅れてしまう。

「どう書けばいいのか分からない」まま時間だけが過ぎ、  
周囲から指摘され、自分でも強い自己嫌悪を感じていました。

発達特性の影響もあり、  
思考を整理して言語化することに大きな負担を感じていた私は、  
**思考整理と言語化を助けるツールが必要だ** と強く感じるようになりました。


### 2-2. その体験の中で一番つらかった瞬間

一番つらかったのは、  
業務内容を思い出せているのに日報が書けず、  
周囲が退勤する中で一人だけ残っていたときです。

原因が分からないまま何度も書き直し、  
ただ焦りだけが積み重なっていきました。

この経験から、  
「能力の問題ではなく、思考を整理する手段が足りなかったのではないか」  
と考えるようになりました。

同じように、頭の中では理解しているのに言葉にできず苦しんでいる人の助けになるツールを作りたい。  
それが **MindOutline** を考えたきっかけです。

## 3. 課題の整理（表面的になっていないか）

### 3-1. 表に見えている困りごと

日報を書こうとしても何を書けばよいのか分からず、  
手が止まり、文章を書くのに極端に時間がかかってしまうことです。

### 3-2. なぜそれが困るのか？

日報に時間がかかることは、単なる「文章作成の遅さ」ではなく、次のような影響を生んでいました。

- 業務時間が延びる
- 周囲に迷惑をかけているという罪悪感
- 「自分だけできない」という感覚による自己肯定感の低下
- 仕事への不安や恐怖の増大

さらに、

- 相談ができない
- 考えを伝えられない
- 振り返りができない

という二次的な問題にもつながり、  
仕事や日常生活全体に影響を及ぼしていました。

### 3-3. 本当に解決したい課題は何か

本当に解決したい課題は、

** 頭の中にある情報や考えを構造化し、言葉として外に出せるようにすること**です。

これは単なる文章生成ではなく、  
「思考整理のプロセス」そのものを支援することだと考えています。

私はこれまで、

- ポモドーロタイマーで作業の見通しを立てる
- 優先順位ToDoアプリで行動の順番を整理する

といった形で、自分の困りごとを一つずつ分解してきました。

本アプリでは、  
断片的なメモ → 構造化 → 文章化 という流れを設計することで、  
言語化に悩む人の負担を減らすことを目指します。

**このアプリは、自分自身が毎日使い続けられることを前提に設計します。**

## 5. 既存サービス・競合調査

### 5-1. 似たサービスの調査

①メモアプリ系

**Notion**  
（URL：https://www.notion.com/）

- メモ・DB・タスク管理を統合した多機能ワークスペース
- 自由度が高く柔軟な情報整理が可能

→ しかし「何を書けばよいか分からない状態」では入力が止まりやすい。

**Evernote**  
（URL：https://evernote.com/）

- ノート作成、検索、タグ整理が可能

→ 情報の蓄積には強いが、思考整理や文章生成の導線は弱い。

②マインドマップ系

**MindMeister**  
（URL：https://www.mindmeister.com/）

- 思考を視覚的に整理できる

→ 構造化はできるが、文章作成までの支援がない

**XMind**  
（URL：https://www.xmind.net/）

- アイデア整理に特化

→ 構造から文章への変換は手動になる

③AIチャット系

**ChatGPT** （提供：OpenAI）  
（URL：https://chat.openai.com/）

- 文章生成が可能

→ 何を入力するかを自分で整理する必要がある

### 5-2. 既存サービスがある中で、それでも作る理由

これらのサービスを試した結果、  
私の課題は解決しませんでした。

理由は、どのサービスも

- 自由度が高すぎる
- 入力内容を自分で整理する必要がある
- 書き出しで止まってしまう

という問題があったためです。

私が本当に必要としていたのは、  
思考がまとまっていなくても一歩ずつ進める導線を持つツールでした。

### 5-3. アプリの簡易説明と差別化ポイント

本アプリは、以下の3点で既存サービスと異なります。

**① 入力のしやすさ**  
キーワードメモから始められるため、書き出しの心理的負担が小さい。

**② 思考整理の導線**  
ツリー構造によって、断片メモを自然に構造化できる。

**③ 文章化まで一気通貫**  
単一メモからAIが短文を生成し、日報や相談文のたたき台を作る。

「メモ → 構造化 → 文章生成」を一つの流れで支援する点が本アプリの特徴です。

## 7. このアプリで実現すること

### 7-1. なぜツリー構造なのか

人は、最初から文章で考えることができません。  
多くの場合、

- キーワード
- 思いついた断片
- 箇条書き

といった形で考え始めます。

ツリー構造を使うことで、

- 思考を段階的に整理できる
- 大枠 → 詳細の流れを自然に作れる
- 書く順番に迷わなくなる

という効果があります。

これは、日報や相談文を書くときに「何から書けばいいか分からない」状態を解消するために必要な仕組みです。

### 7-2. なぜAI文章生成なのか

思考が整理できても、文章にする段階で止まってしまう人は多くいます。

AI文章生成を使うことで、

- 断片メモから文章のたたき台を作れる
- 表現に迷う時間を減らせる
- 「ゼロから書く」心理的ハードルを下げられる

という効果があります。

AIは完成品を書くためではなく、  
最初の一歩を助ける補助ツールとして使います。

### 7-3. MVPで作る機能

最小限の価値として、思考整理から文章生成までの一連の流れを実装します。

- ユーザー登録・ログイン（Devise）
- メモCRUD
- ツリー構造表示（ancestry）
- 単一メモからAI文章生成（短文）

この段階で、  
「キーワード → 構造化 → 短文生成」  
の体験を完成させます。  

### 7-4. 本リリースで作る機能

MVPの体験をより実用的・継続的にするため、次の機能を追加予定です。

**① AI機能の強化**

- 複数メモから文章生成
- 文体変更（丁寧・報告用・相談用など）
- 生成履歴の保存

**② 思考整理の強化**

- メモ検索・タグ機能
- 並び替え・優先順位表示
- Markdown / PDFエクスポート

**③ 継続利用の支援**

- 週次振り返りレポート生成
- PomodoroTimerとの連携による作業ログ分析
- 「今日の3タスク」表示モード

### 7-5.最終的に目指す姿

このアプリは、  
単なるメモアプリではなく、

**思考整理 → 行動 → 言語化 → 振り返り**

を支援する、継続型の自己理解ツールへ発展させたいと考えています。

## 8. このアプリの懸念点とその対策

### 懸念点①：既存サービスとの差別化が弱く見える可能性

- 何が問題になりそうか：  
　「メモアプリ＋ツリー構造」だけでは既存サービスとの差別化が弱いと見られる可能性。

- なぜそう思うか：  
　Notion や XMind などにも構造化機能があり、機能面だけで見ると類似しているため。

- そのために考えている対策：  
　機能の多さではなく「言語化で止まる瞬間を助ける導線」と「AI短文生成までを一体化した体験」に焦点を当て、READMEやデモ動画で体験価値を明確に伝える。

### 懸念点②：ツリー構造の実装難易度

- 何が問題になりそうか：  
　parent_idによる階層構造の実装でバグや設計ミスが発生する可能性。

- なぜそう思うか：  
　自己参照関連は、再帰処理やdependent設定の誤りによりデータ整合性が崩れるリスクがあるため。

- そのために考えている対策：  
　まずフラットCRUDを完成 → 小さなツリーで検証 → gem（ancestry）導入の順で段階的に実装する。  
　テストコードで削除・移動・循環参照を確認する。

### 懸念点③：AI機能の技術的・コスト的リスク

- 何が問題になりそうか：  
　APIエラー、レスポンス整形、課金管理などの問題が発生する可能性。

- なぜそう思うか：  
　外部APIはレート制限や仕様変更などの外部依存リスクがあるため。

- そのために考えている対策：  
　MVPでは「単一メモから短文生成」に限定し、
  - 入力文字数制限  
  - 生成回数制限  
  - エラーハンドリング
  
  を実装する。  
      本リリースで複数メモ生成や履歴管理を追加する。

### 懸念点④：ターゲットが狭すぎる可能性

- 何が問題になりそうか：  
　「言語化に悩む人」というニッチな層に限定される可能性。

- なぜそう思うか：  
　一般的なタスク管理より用途が限定的に見えるため。

- そのために考えている対策：  
　まず自分が継続利用し、  
　日報・相談文・議事録・振り返りなど用途を広げ、  
　READMEで具体的な使用例を提示する。

### 懸念点⑤：開発途中で機能を広げすぎてしまう可能性

- 何が問題になりそうか：  
　AI改善やUI変更を途中で追加し、MVP完成が遅れる可能性。

- なぜそう思うか：  
　過去の個人開発でも機能追加によりスケジュールが崩れた経験があるため。

- そのために考えている対策：  
　MVPは  

  - ログイン  
  - メモCRUD
  - ツリー表示
  - 単一メモAI生成

  までと定義し、それ以外はIssueに分離して管理する。

### 懸念点⑥：ユーザーが継続して使わない可能性

- 何が問題になりそうか：  
　入力が面倒、効果を感じられないなどで継続利用されない可能性。

- なぜそう思うか：  
　既存メモアプリを使い続けられなかった自分の経験があるため。

- そのために考えている対策：
   - キーワード入力だけでも使えるUI
   - AIで最初の一文を生成
   - 「今日の3タスク」など最小表示モード

  まず自分が毎日使うことを基準に改善する。

### 懸念点⑦：開発環境トラブル（Docker / バージョン差異）

- 何が問題になりそうか：  
　Dockerを途中から導入した場合、ローカル環境との差異により予期せぬエラーが発生し、開発やデプロイに時間を要する可能性。

- なぜそう思うか：  
　過去の開発経験でも、RubyやRailsのバージョン差異、PostgreSQLの設定違いなどにより環境依存のエラーが発生したことがあるため。

- そのために考えている対策：  
　開発初期段階から Docker を導入し、  
　Ruby・Rails・PostgreSQLのバージョンを明示的に固定する。

  具体的には：

   - Ruby 3.2系で固定
   - Rails 7.0系で固定
   - PostgreSQL 15系で固定
   - GemのバージョンをGemfile.lockで管理

  これにより、環境差異によるトラブルを最小限に抑え、  
デプロイやトラブルシューティングの時間を削減する。

## 10. 技術スタック（手段としての技術）

### 10-1. 使用予定の技術

- 開発環境：Docker / docker-compose
   - 開発言語：Ruby 3.2.3
   - 開発フレームワーク：Rails 7.0.8
   - DB：PostgreSQL 14
   - その他フレームワーク：TailwindCSS
- デプロイ先：Render
- 使用予定ライブラリ：
   - Devise（ログイン機能で使用）
   - ancestry（ツリー構造を操作するのに使用）
   - ruby-openai（AIによる文章生成で使用）

### 10-2. 技術選定の理由

本アプリは、思考整理から文章化までを支援するWebアプリであり、  
RUNTEQカリキュラムで学習した技術を基盤として確実にMVPを完成させることを重視しました。

■ Rails
RUNTEQのカリキュラムで最も経験を積んだフレームワークであり、  
認証・CRUD・API連携など卒業制作に必要な機能を安定して実装できるため選択しました。

■ TailwindCSS  
CSSを書かなくてもよく、レイアウト修正が簡単なため、Railsとも相性がいいと判断し採用しました。

■ PostgreSQL  
Railsとの相性が良く、RenderなどのPaaSでも標準的に利用できるため採用しました。

■ Devise  
ログイン機能を安全かつ短期間で実装できるため採用しました。
認証処理の自作によるバグを防ぎ、MVPの完成を優先するためです。

■ ancestry  
メモの親子関係による思考整理を実現するため採用しました。
自己参照テーブルより実装がシンプルで、理解しながらツリー構造を扱える点を重視しました。
closure_treeも検討しましたが、機能が豊富な分ブラックボックス化しやすく、学習目的と保守性を考えancestryを選択しました。

■ ruby-openai  
本リリースで予定している文章生成機能のために採用予定です。
RUNTEQカリキュラムのRails×OpenAI APIの内容を応用し、
断片メモから文章を生成する機能に挑戦したいと考えています。
APIエラー時は生成なしでもメモ機能が使える設計にし、コスト制御のため1日回数制限を設ける予定です。

■ Render  
無料枠でRails＋PostgreSQLを簡単にデプロイできるため選択しました。
卒業制作ではインフラよりもUXと機能完成度を重視し、
まず確実に公開できる環境を優先します。

■ Docker  
本アプリでは、開発初期段階からDockerを導入しています。

**採用理由**

- ローカル環境と本番環境の差異を最小限に抑えるため
- Ruby・Rails・PostgreSQLのバージョンを明示的に固定するため
- 将来的なチーム開発や再現性のある環境構築を見据えているため

途中からDocker化すると環境差異によるトラブルが発生しやすいため、  
最初からコンテナベースで開発を進める方針としました。

また、環境構築に時間を取られるのではなく、  
本アプリの価値である「思考整理体験」の実装に時間を使うことを優先しています。

### 10-3. 今回のチャレンジ点

- ツリー構造データの設計と実装
- OpenAI APIとの連携による文章生成
- 自分の実体験をもとにしたUX設計
- 毎日使いたくなるシンプルなUI設計

### 10-4. 不安な点とその対策

- ツリー構造の設計ミスによるデータ整合性
- OpenAI API連携時のエラー処理やコスト管理
- MVP期間内に機能を広げすぎてしまうこと

そのため、フェーズを分けて開発し、  
まずはメモCRUDとツリー表示とAI文章機能の完成を最優先とします。
